# 分布式系统负载均衡算法演进与项目展示方案

本文用于 45 分钟左右的分布式操作系统课程汇报准备。主题是“分布式系统负载均衡算法”，重点不是只罗列算法名称，而是说明算法为什么一步步演进，以及这些算法如何映射到当前项目的任务调度框架中。

当前项目已经具备一个适合做算法验证的基本结构：

- `Scheduler` 维护任务队列、Worker 心跳状态和负载均衡决策。
- `Worker` 定期上报 `running_tasks`、`cpu_percent`、`memory_percent`、`max_concurrent_tasks`，并主动调用 `PullTask` 拉取任务。
- `distributed_scheduler/scheduler/load_balancer.py` 是策略扩展点。
- `distributed_scheduler/scheduler/task_store.py` 是队列、Worker 状态、任务状态流转的核心位置。
- `config/scheduler.yaml` 通过 `strategy` 切换策略。

因此，汇报中可以把当前项目作为“微型分布式任务调度系统”，用它展示从静态策略到动态策略、从中心式调度到分布式调度、从单指标到多资源指标的演进。

## 一、负载均衡算法的演进主线

负载均衡算法的历史大致可以分为五个阶段：

1. **静态分配阶段**：早期分布式系统规模较小，节点性能相近，任务也相对规则。算法主要依赖固定规则，例如随机、轮询、按权重轮询、静态哈希。优点是实现简单，缺点是不感知真实负载。
2. **动态状态感知阶段**：随着 Web 服务、服务器集群和 TCP 连接型服务出现，调度器开始跟踪连接数、队列长度、响应时间等运行状态，典型算法包括最少连接、加权最少连接、最短期望延迟。
3. **会话亲和与缓存友好阶段**：Web 会话、缓存系统、分布式存储要求“相同 key 尽量落到同一节点”，于是出现源地址哈希、目标哈希、一致性哈希、Maglev 等算法。
4. **大规模随机化与去中心化阶段**：当服务器规模从几十台增长到成百上千台，调度器很难维护全局精确状态。随机采样、Power of Two Choices、Work Stealing、Join-Idle-Queue 等算法开始强调“少量状态也能得到接近最优的效果”。
5. **云原生与多资源调度阶段**：容器、批处理、大数据和多租户集群要求同时考虑 CPU、内存、数据本地性、公平性、故障、长尾任务和拓扑结构。Kubernetes、Mesos、Spark、Envoy 等系统把负载均衡扩展为资源评分、数据本地性、推测执行、服务网格策略组合。

汇报时可以把这条线索概括为：

> 从“不看负载的平均分配”，演进到“看运行状态的动态分配”，再演进到“兼顾亲和性、局部信息、多资源和故障恢复的综合调度”。

## 二、历史进程中的代表算法概览

| 序号 | 算法 | 大致阶段 | 核心思想 | 适合展示的问题 |
| --- | --- | --- | --- | --- |
| 1 | 静态分区 / 静态映射 | 早期分布式系统 | 任务按固定规则绑定到节点 | 简单但无法适应负载变化 |
| 2 | Random 随机分配 | 早期和大规模系统基础线 | 随机选择一个可用节点 | 无状态、低开销，但波动大 |
| 3 | Round Robin 轮询 | DNS、LVS、反向代理 | 按顺序循环选择节点 | 请求数均匀，不等于负载均匀 |
| 4 | Weighted Round Robin 加权轮询 | 异构服务器集群 | 高权重节点获得更多任务 | 处理不同节点能力 |
| 5 | Least Connections / JSQ 最少连接或最短队列 | Web/L4/L7 负载均衡 | 选择连接数或队列长度最小节点 | 长短任务混合时优于轮询 |
| 6 | Weighted Least Connections 加权最少连接 | LVS/IPVS 等 | 比较连接数与权重的比例 | 异构节点下的动态负载 |
| 7 | Shortest Expected Delay 最短期望延迟 | LVS/IPVS 等 | 估计 `(当前负载+1)/服务能力` | 从连接数走向排队延迟估计 |
| 8 | Least Response Time / Least Time 最短响应时间 | NGINX Plus、应用网关 | 选择平均响应时间较低的节点 | 体现真实服务耗时反馈 |
| 9 | Source/IP Hash 源地址哈希 | 会话保持、缓存 | 同一来源或 key 固定映射到同一节点 | 会话亲和与负载均衡的冲突 |
| 10 | Consistent Hashing 一致性哈希 | 分布式缓存、存储 | 节点变动时只迁移少量 key | 扩缩容时减少扰动 |
| 11 | Power of Two Choices 二选一随机采样 | 大规模随机化调度 | 随机抽两个节点，选更空闲者 | 低开销接近全局最小队列 |
| 12 | Work Sharing 发送方发起迁移 | 自适应负载共享 | 过载节点主动把任务转出 | 高负载节点主动卸载 |
| 13 | Work Stealing 工作窃取 | 并行运行时、分布式任务系统 | 空闲节点主动从忙节点拿任务 | 适合不规则任务和长尾 |
| 14 | Join-Idle-Queue 空闲队列 | 大规模 Web 前端 | 空闲服务器提前登记，调度器优先派给空闲节点 | 多调度器下减少查询开销 |
| 15 | Delay Scheduling 延迟调度 | Hadoop/Spark | 为数据本地性短暂等待合适节点 | 吞吐、公平、本地性之间权衡 |
| 16 | Speculative Execution 推测执行 | MapReduce 等 | 长尾任务复制执行，先完成者生效 | 处理慢节点和尾延迟 |
| 17 | Multi-resource Scoring / DRF 多资源评分与公平 | Mesos/Kubernetes | 同时考虑 CPU、内存、资源请求和公平性 | 从单负载指标走向资源调度 |
| 18 | Maglev / Lookup Table 一致性负载均衡 | Google L4 负载均衡 | 构造稳定查找表，兼顾均匀性和故障扰动 | 大规模软件负载均衡工程化 |

如果课程时间有限，可以重点讲 15 个：保留 1 到 17，Maglev 作为一致性哈希的工程化补充；如果希望更贴近现代系统，可以保留 Maglev，合并 Work Sharing 和 Work Stealing。

## 三、逐个算法说明与项目落地思路

### 1. 静态分区 / 静态映射

**实现逻辑**

静态分区把任务按照固定规则分给节点，例如：

- `task_id % worker_count`
- 按任务类型固定分配：CPU 任务给 `worker-cpu-*`，文本任务给 `worker-text-*`
- 按租户或用户固定分配：`tenant_id -> worker_id`

这种方法不需要实时心跳，不需要计算负载，也几乎没有调度开销。缺点是节点性能变化、任务耗时变化、热点 key 出现时，系统无法自动纠偏。

**在当前项目中的实现方式**

可以在 `LoadBalancer.choose()` 中增加 `static_mapping` 策略，但它需要知道当前任务的信息。当前 `choose()` 只接收 `workers`，因此需要把接口调整为：

```python
choose(workers: list[WorkerRecord], task: TaskRecord | None = None)
```

然后用 `task.task_type`、`task.name` 或 `task_id` 做固定映射。若不修改接口，也可以做一个简化版本：按 `worker_id` 排序后始终选择第一个可用节点，但这只能展示“静态偏置”，不是真正的任务分区。

**展示差异**

设计一个热点任务场景：大量 `fibonacci` 任务都映射到同一类 worker。展示静态分区在节点能力变化时会让某些 worker 长期繁忙，而其他 worker 空闲。

### 2. Random 随机分配

**实现逻辑**

每次从可用 Worker 列表中随机选择一个节点：

```python
random.choice(workers)
```

随机策略不维护游标，不依赖实时负载，天然适合多个调度器并发工作的环境，因为不同调度器不会因为轮询游标不一致而产生复杂协调。

缺点是短时间内波动明显，可能连续命中同一个 Worker；当任务耗时差异很大时，随机分配可能形成长队列。

**在当前项目中的实现方式**

在 `load_balancer.py` 中增加：

```python
if self.strategy == "random":
    return random.choice(ordered_workers)
```

配置文件 `config/scheduler.yaml` 中增加 `strategy: "random"`。这个算法不需要修改 `WorkerRecord`。

**展示差异**

提交 30 个相同的 `sleep 1` 任务，比较 `random` 和 `round_robin` 的每个 worker 执行数量。随机策略长期趋于均衡，但短期分布更抖动。

### 3. Round Robin 轮询

**实现逻辑**

轮询策略维护一个指针，每次选择下一个 Worker：

```python
index = current_index % len(workers)
current_index += 1
return workers[index]
```

轮询的关键假设是：节点能力接近，任务成本接近。它保证“任务数量”比较平均，但不保证“工作量”比较平均。

DNS Round Robin、LVS/IPVS、NGINX 等都支持类似方法。DNS 层轮询的粒度较粗，受 DNS 缓存影响；L4/L7 代理可以按连接或请求做更细粒度分配。

**在当前项目中的实现方式**

当前项目已经实现 `round_robin`，位置在 `distributed_scheduler/scheduler/load_balancer.py`。它对 `worker_id` 排序后使用 `_round_robin_index` 轮转，这样演示结果稳定。

**展示差异**

启动三个 Worker，提交交替的 `sleep 1` 和 `sleep 8` 任务。轮询会让任务数量均衡，但某些 Worker 被长任务拖住，导致完成时间不均衡。

### 4. Weighted Round Robin 加权轮询

**实现逻辑**

加权轮询为每个 Worker 设置权重，权重越高，分到的任务越多。例如权重 `4:2:1` 可以让三个节点在一个周期里大致按这个比例接收任务。

常见实现方式有两类：

- 展开序列：`[A, A, A, A, B, B, C]` 循环选择，简单但权重大时序列长。
- 平滑加权轮询：维护每个节点的当前权重，每次选当前权重最高者，再减去总权重，分布更平滑。

**在当前项目中的实现方式**

当前 `WorkerRecord` 没有显式 `weight` 字段，但可以先用 `max_concurrent_tasks` 作为权重。能力强的 Worker 启动时传入更大的 `--max-concurrent-tasks`，调度器就能给它更多任务。

可选实现：

- 在 `WorkerRecord` 增加 `weight`，从心跳或配置读取。
- 在 `LoadBalancer` 中维护 `current_weights: dict[str, int]`。
- 选择时跳过没有容量或心跳过期的 Worker。

**展示差异**

启动：

```bash
python -m distributed_scheduler.worker.worker_node --worker-id worker-small --max-concurrent-tasks 1
python -m distributed_scheduler.worker.worker_node --worker-id worker-big --max-concurrent-tasks 4
```

提交 50 个短任务。展示普通轮询会低估大节点能力，而加权轮询能让大节点承担更多任务。

### 5. Least Connections / Join Shortest Queue 最少连接或最短队列

**实现逻辑**

最少连接选择当前活跃连接数最少的节点；任务系统中对应的是最短队列或最少运行任务数：

```python
min(workers, key=lambda worker: worker.running_tasks)
```

这类策略开始利用系统状态，是从静态负载均衡走向动态负载均衡的关键一步。它适合长短请求混合的场景，因为长任务占用节点时，该节点的连接数或运行任务数会持续偏高。

**在当前项目中的实现方式**

当前项目已经实现 `least_loaded`，排序依据是：

1. `running_tasks`
2. `cpu_percent`
3. `memory_percent`
4. `worker_id`

它本质上就是最少运行任务数策略，并用 CPU、内存作为并列时的 tie-breaker。

**展示差异**

提交一组混合任务：

- `sleep 10` 长任务 5 个
- `sleep 1` 短任务 30 个

对比 `round_robin` 和 `least_loaded` 的平均完成时间、最大完成时间和每个 Worker 的 `running_tasks` 曲线。

### 6. Weighted Least Connections 加权最少连接

**实现逻辑**

加权最少连接不是简单比较 `running_tasks`，而是比较“当前负载与节点能力的比例”：

```text
score = active_connections / weight
```

选择 `score` 最低的节点。LVS/IPVS 文档中也采用类似思想，用连接数和权重比例做调度。

这个算法解决了最少连接对异构节点不友好的问题。一个 8 核节点和一个 2 核节点都有 2 个连接时，显然 8 核节点更有余量。

**在当前项目中的实现方式**

可以使用：

```python
capacity = max(worker.max_concurrent_tasks, 1)
score = worker.running_tasks / capacity
```

策略名可以叫 `weighted_least_loaded`。不需要改 protobuf，因为 `max_concurrent_tasks` 已经由心跳上报。

**展示差异**

启动不同并发能力的 Worker，提交大量耗时接近的任务。对比 `least_loaded` 和 `weighted_least_loaded`：

- `least_loaded` 倾向让所有 Worker 的运行任务数相近。
- `weighted_least_loaded` 倾向让所有 Worker 的并发占用率相近。

### 7. Shortest Expected Delay 最短期望延迟

**实现逻辑**

最短期望延迟进一步把“队列长度”和“服务能力”组合成延迟估计。LVS/IPVS 的 SED 思想可写为：

```text
expected_delay = (connections + 1) / service_rate
```

`+1` 表示新任务加入后的影响，`service_rate` 可以用权重、CPU 核数、历史吞吐量估计。

**在当前项目中的实现方式**

可用 `max_concurrent_tasks` 近似服务能力：

```python
score = (worker.running_tasks + 1) / worker.max_concurrent_tasks
```

如果要更真实，可以为 `WorkerRecord` 增加 `completed_tasks` 和 `avg_duration_ms`，用历史吞吐量估计 `service_rate`。

**展示差异**

设置 `worker-fast` 的 `max_concurrent_tasks=4`，`worker-slow` 的 `max_concurrent_tasks=1`。提交相同任务，展示 SED 比普通最少连接更愿意选择高能力节点，但又不会完全压满高能力节点。

### 8. Least Response Time / Least Time 最短响应时间

**实现逻辑**

最短响应时间类策略不只看当前连接数，还看历史请求延迟。例如 NGINX Plus 的 Least Time 会考虑平均延迟和活动连接数。抽象到任务系统，可以维护每个 Worker 的历史完成耗时：

```text
score = ewma_task_duration_ms * alpha + running_tasks * beta
```

其中 EWMA 是指数滑动平均，能让新数据逐步影响评分。

**在当前项目中的实现方式**

当前 `ReportTaskResult` 已经能知道 `started_at_unix_ms` 和 `finished_at_unix_ms`。可在 `TaskStore.report_task_result()` 中更新 Worker 的：

- `avg_duration_ms`
- `completed_count`
- `last_error_count`

然后在 `LoadBalancer` 中新增 `least_response_time` 策略。

**展示差异**

人为让某个 Worker 执行更慢，例如给它较小 `max_concurrent_tasks` 或在任务执行器中为某个 worker 注入额外 sleep。展示该策略会逐渐减少分配给慢节点的任务，而轮询不会。

### 9. Source/IP Hash 源地址哈希

**实现逻辑**

源地址哈希用请求来源、用户 ID、会话 ID 或业务 key 计算哈希，然后映射到节点：

```text
worker_index = hash(client_ip_or_key) % worker_count
```

它的目标不是绝对均衡，而是会话保持。相同用户持续访问同一节点，可以减少登录状态、缓存数据、连接上下文的迁移。

缺点是热点用户或热点 key 会压垮单个节点；节点数量变化时，普通取模哈希会导致大量 key 重新映射。

**在当前项目中的实现方式**

当前任务没有 `client_ip` 或 `affinity_key` 字段，可以用以下方式模拟：

- 用 `TaskRecord.name` 的前缀作为用户 ID，例如 `user-a-task-001`。
- 更规范地在 `SubmitTaskRequest` 增加 `affinity_key` 字段。
- 在 `LoadBalancer.choose()` 中根据 `task.affinity_key` 选择 Worker。

由于 `choose()` 当前没有 task 参数，需要调整接口。

**展示差异**

构造 3 个用户：

- `user-a` 提交 80% 任务。
- `user-b`、`user-c` 各提交 10% 任务。

展示源哈希能保持用户亲和，但热点用户会造成负载倾斜。

### 10. Consistent Hashing 一致性哈希

**实现逻辑**

一致性哈希把 Worker 和任务 key 都映射到一个环上。任务落到顺时针方向遇到的第一个 Worker。加入或删除节点时，只影响环上相邻的一部分 key，而不是所有 key。

为改善均匀性，实际实现通常使用虚拟节点：

```text
hash(worker-1#0), hash(worker-1#1), ..., hash(worker-1#99)
```

同一真实 Worker 在环上有多个位置，减少分布偏斜。

**在当前项目中的实现方式**

需要在 `LoadBalancer` 中维护哈希环：

- 输入：当前存活且有容量的 Worker。
- 构造：为每个 Worker 生成多个虚拟节点。
- 选择：对 `task.affinity_key` 或 `task_id` 哈希，二分查找第一个大于等于该值的虚拟节点。
- 容量保护：如果目标 Worker 没容量，则继续顺时针找下一个可用 Worker。

该策略同样需要 `choose(workers, task)`。

**展示差异**

先启动 3 个 Worker，提交一批带 key 的任务，记录 key 到 worker 的映射。再加入第 4 个 Worker，对比：

- 普通 `hash(key) % n` 会大量重映射。
- 一致性哈希只迁移部分 key。

### 11. Power of Two Choices 二选一随机采样

**实现逻辑**

Power of Two Choices 的核心非常简单：

1. 随机抽取两个候选 Worker。
2. 比较这两个 Worker 的负载。
3. 把任务交给更空闲的那个。

```python
a, b = random.sample(workers, 2)
return min([a, b], key=lambda worker: worker.running_tasks)
```

它的意义在于：不需要查询全局所有节点，只用两个样本就能显著降低最大负载。在大规模系统中，这比全局最短队列更便宜，也比纯随机更稳定。

**在当前项目中的实现方式**

在 `LoadBalancer` 中增加 `power_of_two`。如果候选 Worker 少于 2 个，直接退化为随机或最少负载。

可以把比较指标替换为：

- `running_tasks`
- `load_score()`
- `(running_tasks + 1) / max_concurrent_tasks`

**展示差异**

增加一个统计脚本，重复提交 100 或 500 个短任务，比较：

- `random`
- `round_robin`
- `power_of_two`
- `least_loaded`

重点观察最大 Worker 任务数、负载标准差和平均完成时间。`power_of_two` 的效果通常会明显优于随机，同时开销小于全局最少负载。

### 12. Work Sharing 发送方发起迁移

**实现逻辑**

Work Sharing 是“过载节点主动转出任务”。当某个节点发现自己的队列过长时，主动寻找轻载节点，把尚未执行的任务迁移出去。

典型过程：

1. Worker 判断自己超过阈值，例如 `queue_length > high_watermark`。
2. 它向调度器或其他 Worker 查询轻载节点。
3. 把未开始执行的任务转移给轻载节点。

它适合任务从各个节点本地产生的系统，比如早期工作站集群或分布式执行环境。

**在当前项目中的实现方式**

当前项目是中心队列，不存在 Worker 本地待执行队列，所以要展示 Work Sharing 需要扩展：

- 让 Scheduler 可以预分配多个任务到 Worker 本地队列。
- Worker 心跳上报 `local_queue_length`。
- 当 Worker 本地队列过长时，调用新 RPC，例如 `ReturnTask` 或 `RebalanceTask`，把未开始任务交回 Scheduler。

课程展示可以只做概念模拟：在文档或图中展示它与当前“中心队列 + Worker 拉取”的差异。

**展示差异**

构造一个 Worker 被一次性分配很多任务的场景。Work Sharing 能把排队任务转出，降低该 Worker 的尾部完成时间。

### 13. Work Stealing 工作窃取

**实现逻辑**

Work Stealing 与 Work Sharing 的方向相反：空闲节点主动寻找忙节点，从忙节点队列中“偷”任务。经典并行运行时中，每个 Worker 有本地双端队列，自己从一端取任务，空闲者从其他 Worker 的另一端偷任务。

基本过程：

1. Worker 本地队列为空。
2. 随机选择一个 victim Worker。
3. 如果 victim 有未开始任务，就迁移一部分到自己。

它适合不规则任务，因为系统不必预先准确估计每个任务耗时。

**在当前项目中的实现方式**

当前 Worker 已经是主动拉取任务，这和“空闲者主动要工作”的思想相近，但仍然由 Scheduler 中心队列决定，不是真正从其他 Worker 偷。

若要实现真实 Work Stealing，需要：

- 为每个 Worker 增加本地待执行队列。
- Scheduler 记录每个 Worker 的本地队列长度。
- 新增 `StealTask` 或由 Scheduler 在 `PullTask` 时从忙 Worker 的未执行队列迁移任务。

简化展示方式：把当前 `PullTask` 解释为“中心化的拉取式调度”，再对比真正 Work Stealing 是“从其他 Worker 队列拉取”。

**展示差异**

提交一批耗时高度不均匀的任务，例如随机 `sleep 1` 到 `sleep 12`。工作窃取能让先完成任务的 Worker 继续处理其他节点队列中的剩余任务，减少整体 makespan。

### 14. Join-Idle-Queue 空闲队列

**实现逻辑**

Join-Idle-Queue 解决大规模多调度器场景中的状态查询开销。它不要求调度器在每个任务到来时查询所有 Worker，而是让空闲 Worker 提前把自己登记到一个空闲队列中。

过程：

1. Worker 空闲时向某个 dispatcher 注册一个 idle token。
2. 新任务到来时，dispatcher 优先从 idle queue 中取 token。
3. 如果没有 idle token，再退化为随机或其他策略。

它把“发现空闲节点”从任务到达关键路径中移出去，适合大规模 Web 前端。

**在当前项目中的实现方式**

当前项目只有一个 Scheduler，也维护全局 Worker 状态，因此不迫切需要 JIQ。但可以模拟：

- 在 `TaskStore` 中维护 `_idle_worker_ids: deque[str]`。
- Worker 心跳时，如果 `running_tasks == 0`，加入空闲队列。
- `PullTask` 或中心派发时优先选择 idle queue 中的 Worker。
- 注意去重和 Worker 失效清理。

**展示差异**

重点不是单机性能，而是讲解“多调度器时全局状态难以同步”。课堂可用架构图对比：

- JSQ：每次任务到来都要查询或维护所有 Worker 队列长度。
- JIQ：空闲状态提前登记，任务到来时 O(1) 派发。

### 15. Delay Scheduling 延迟调度

**实现逻辑**

Delay Scheduling 最初用于解决大数据集群中的公平性和数据本地性冲突。如果当前最公平的任务不能在有本地数据的节点上运行，调度器不是立刻把它放到任意节点，而是短暂等待，让它有机会等到本地节点空闲。

抽象逻辑：

```text
如果首选 Worker 可用：立即调度
否则，如果等待时间未超过阈值：暂不调度该任务，看看后续机会
否则：放宽本地性要求，调度到任意可用 Worker
```

它牺牲一点等待时间，换取数据本地性、缓存命中和整体吞吐。

**在当前项目中的实现方式**

当前任务没有数据位置概念，可以模拟：

- 给 `SubmitTaskRequest` 增加 `preferred_worker_ids` 或 `data_location`。
- `TaskRecord` 记录 `locality_wait_started_at`。
- `_pop_next_queued_task_locked()` 不能只弹 FIFO 第一个任务，而要能扫描队列，跳过正在等待本地性的任务。
- 超过等待阈值后允许分配给非首选 Worker。

**展示差异**

设置某些任务偏好 `worker-1`，但 `worker-1` 短时间繁忙。对比：

- 立即调度：等待少，但本地性差。
- 延迟调度：部分任务稍等，最终更多任务在首选节点执行。

展示指标可以是本地性命中率、平均等待时间、整体完成时间。

### 16. Speculative Execution 推测执行

**实现逻辑**

推测执行用于处理长尾任务和慢节点。系统发现某个任务运行时间明显超过同类任务时，会把同一个任务复制到另一个 Worker 上执行，谁先完成就采用谁的结果，另一个副本取消或忽略。

MapReduce 中的 backup tasks 就是代表性机制。它不是传统意义上的“选择一个节点”，而是通过冗余执行降低尾延迟。

**在当前项目中的实现方式**

当前项目已经有超时重入队逻辑，但那是失败恢复，不是推测执行。要实现推测执行，需要：

- `TaskRecord` 支持多个尝试记录，例如 `attempts: list[TaskAttempt]`。
- Scheduler 周期性检查运行时间超过阈值的任务。
- 为慢任务创建 speculative attempt，分配给另一个 Worker。
- `ReportTaskResult` 接收第一个成功结果，后续结果标记为 ignored。

**展示差异**

让一个 Worker 人为变慢，提交多个 `sleep` 或 `fibonacci` 任务。对比：

- 无推测执行：整体完成时间被最慢任务拖住。
- 推测执行：多消耗一些资源，但 p95/p99 完成时间下降。

### 17. Multi-resource Scoring / DRF 多资源评分与公平

**实现逻辑**

现代集群不只看“几个任务正在跑”，还要看 CPU、内存、磁盘、网络、GPU、租户公平性等多个维度。Kubernetes 调度器会先过滤不满足资源请求的节点，再给节点打分；Mesos/DRF 则强调多资源公平。

当前项目的 `weighted_score` 已经是多资源评分的简化版本：

```text
load_score =
    cpu_percent * 0.4
  + memory_percent * 0.3
  + concurrency_ratio * 0.3
```

更完整的多资源调度会考虑“任务需要什么资源”，而不只是“节点当前有多忙”。

**在当前项目中的实现方式**

可扩展方向：

- 在 `SubmitTaskRequest` 中增加 `requested_cpu`、`requested_memory_mb`。
- Worker 心跳上报 `total_cpu`、`available_memory_mb`、`gpu_count` 等资源。
- Scheduler 先过滤无法满足请求的 Worker，再按加权分数排序。
- 如果要展示公平性，可增加 `tenant_id`，计算每个租户占用的主导资源份额。

**展示差异**

构造三类任务：

- CPU 型：`fibonacci`
- 内存型：可新增模拟任务
- I/O 等待型：`sleep`

展示只看 `running_tasks` 会把不同资源需求混在一起；多资源评分可以让 CPU 压力高的 Worker 少接 CPU 任务，内存压力高的 Worker 少接内存任务。

### 18. Maglev / Lookup Table 一致性负载均衡

**实现逻辑**

Maglev 是 Google 的软件网络负载均衡系统。它用一致性哈希思想构造查找表，让每个负载均衡实例在相同后端集合下得到相同映射。新连接根据五元组哈希到查找表中的一个后端。节点变化时，只有部分连接映射受影响，同时查找速度非常快。

Maglev 更偏网络层连接负载均衡，但对课程汇报有价值，因为它体现了现代系统对以下目标的组合：

- 高性能查表
- 多个负载均衡器之间结果一致
- 后端故障时减少连接扰动
- 软件方式替代专用硬件

**在当前项目中的实现方式**

课程项目可以把它作为一致性哈希的工程化版本，不必完整实现。若要模拟：

- 根据 Worker 列表构造固定大小 lookup table。
- 用 `hash(task_id)` 或 `hash(affinity_key)` 得到表下标。
- 查表得到 Worker。
- Worker 变化时重建表，并统计任务 key 的重映射比例。

**展示差异**

对比三种 key 映射：

- `hash % n`
- 一致性哈希环
- Maglev lookup table

展示指标是扩缩容后的 key 迁移比例和每个 Worker 的 key 数量分布。

## 四、如何在当前项目中组织算法实现

虽然本次任务不要求直接添加算法，但汇报时最好说明“如果要实现，应改哪里”。建议把项目实现分成三层：

### 1. 策略选择层

主要位置：`distributed_scheduler/scheduler/load_balancer.py`

适合放：

- `random`
- `round_robin`
- `weighted_round_robin`
- `least_loaded`
- `weighted_least_loaded`
- `shortest_expected_delay`
- `least_response_time`
- `power_of_two`
- `weighted_score`

这些算法的共同点是：给定一组候选 Worker，选出一个 Worker。

### 2. 状态模型层

主要位置：

- `distributed_scheduler/common/models.py`
- `proto/task_scheduler.proto`
- `distributed_scheduler/scheduler/task_store.py`

适合扩展：

- Worker 权重：`weight`
- Worker 历史延迟：`avg_duration_ms`
- Worker 吞吐：`completed_tasks`
- Worker 本地队列长度：`local_queue_length`
- 任务亲和 key：`affinity_key`
- 任务资源需求：`requested_cpu`、`requested_memory_mb`
- 任务首选位置：`preferred_worker_ids`
- 租户 ID：`tenant_id`

没有这些状态，很多高级算法只能做简化模拟。

### 3. 队列和任务生命周期层

主要位置：`distributed_scheduler/scheduler/task_store.py`

适合放：

- Delay Scheduling：需要跳过暂时等本地性的任务。
- Work Stealing / Work Sharing：需要支持 Worker 本地队列和任务迁移。
- Speculative Execution：需要支持一个任务多个 attempt。
- JIQ：需要维护 idle worker token。

这些算法不仅是“选哪个 Worker”，还会改变任务队列结构和任务生命周期。

## 五、如何更好地展示算法之间的差异

### 1. 统一实验指标

建议每次实验记录以下指标：

| 指标 | 含义 | 能展示什么 |
| --- | --- | --- |
| 平均完成时间 | 任务从提交到完成的平均耗时 | 系统总体效率 |
| p95 / p99 完成时间 | 慢任务尾部延迟 | 长尾和慢节点影响 |
| makespan | 全部任务完成总时间 | 批处理场景吞吐 |
| Worker 任务数分布 | 每个 Worker 完成多少任务 | 数量均衡 |
| Worker 忙碌时间 | 每个 Worker 实际工作时长 | 工作量均衡 |
| 负载标准差 | Worker 负载离散程度 | 均衡程度 |
| CPU / 内存均值和峰值 | 资源使用情况 | 多资源策略效果 |
| 本地性命中率 | 任务是否在首选 Worker 执行 | Delay Scheduling 效果 |
| key 重映射比例 | 节点变化后多少 key 改变归属 | 哈希类算法稳定性 |
| 重复执行开销 | 推测执行多跑了多少副本 | 延迟和资源成本权衡 |

当前 CLI 已能查看任务和 Worker 状态。若做进一步展示，可以增加一个独立脚本收集 `ListTasks` 和 `ListWorkers` 的结果，输出 CSV 或图表。

### 2. 设计对比场景

建议准备 6 类实验场景，每类只对比 2 到 4 个算法，避免汇报过散。

| 场景 | 任务设计 | 对比算法 | 预期结论 |
| --- | --- | --- | --- |
| 同构短任务 | 60 个 `sleep 1` | random、round_robin、least_loaded | 轮询简单稳定，随机短期波动 |
| 长短任务混合 | `sleep 1` 和 `sleep 10` 混合 | round_robin、least_loaded、power_of_two | 动态策略减少长任务堆积 |
| 异构 Worker | Worker 并发度 1、2、4 | round_robin、weighted_rr、weighted_least_loaded | 权重能表达节点能力差异 |
| 慢节点 | 某 Worker 人为变慢 | round_robin、least_response_time、speculative | 延迟反馈和推测执行能降低尾延迟 |
| 热点 key | 80% 任务来自同一 key | source_hash、consistent_hash、least_loaded | 亲和性可能牺牲均衡 |
| 数据本地性 | 任务带首选 Worker | least_loaded、delay_scheduling | 等待本地节点可提升本地性 |

### 3. 展示方式建议

适合课堂演示的图包括：

- **时间线图**：算法从静态到动态、从中心式到分布式、从单指标到多指标。
- **Worker 负载柱状图**：每个 Worker 完成任务数、忙碌时间、平均耗时。
- **任务甘特图**：横轴时间，纵轴 Worker，显示任务开始和结束，直观看出堆积和空闲。
- **折线图**：队列长度随时间变化。
- **散点图**：每个任务的完成时间，突出尾延迟。
- **哈希环图**：展示一致性哈希扩容时只有部分 key 迁移。
- **架构对比图**：中心调度、Worker 拉取、Work Stealing、JIQ 的差异。

### 4. 与当前三种策略的衔接

当前项目已有：

- `round_robin`
- `least_loaded`
- `weighted_score`

可以把它们放在演进链条中：

| 当前策略 | 历史位置 | 可说明的问题 |
| --- | --- | --- |
| `round_robin` | 静态平均分配 | 简单、公平，但不感知真实负载 |
| `least_loaded` | 动态状态感知 | 利用 Worker 运行任务数做反馈 |
| `weighted_score` | 多指标资源感知 | 综合 CPU、内存、并发占用率 |

进一步扩展时，可以按以下顺序添加，最适合课程展示：

1. `random`：作为基础对照组。
2. `weighted_least_loaded`：展示异构 Worker。
3. `power_of_two`：展示大规模随机化思想。
4. `least_response_time`：展示历史反馈。
5. `consistent_hash`：展示会话亲和和扩缩容扰动。
6. `speculative_execution`：展示长尾任务处理，但它涉及任务生命周期，复杂度更高。

## 七、可直接用于汇报的总结观点

1. 负载均衡的目标不只是“请求数量平均”，而是让资源利用率、响应时间、尾延迟、故障恢复和扩缩容扰动都处于可接受范围。
2. 轮询类算法简单，但默认任务和节点是同质的；一旦任务耗时和节点能力差异变大，就需要动态反馈。
3. 最少连接、最短队列和最短期望延迟把排队论思想引入调度，但全局精确状态在大规模系统中成本很高。
4. 哈希类算法牺牲一部分实时均衡，换取会话亲和、缓存命中和扩缩容稳定性。
5. Power of Two Choices 说明少量随机采样可以获得接近全局调度的效果，是大规模系统中非常重要的折中。
6. Work Stealing、JIQ 等算法把“谁来寻找负载”从中心调度器转移到 Worker 或空闲队列，降低中心瓶颈。
7. 现代调度器通常不是单一算法，而是过滤、打分、亲和性、重试、推测执行和公平性策略的组合。
8. 当前项目的 `round_robin -> least_loaded -> weighted_score` 已经能展示负载均衡算法从简单规则到资源感知的演进。
